# 禁止事项详解

## 禁止的输出类型

### 1. 平庸的按文件逐个总结
❌ **错误示例**：
```markdown
## 项目结构
- src/main.js: 主入口文件
- src/utils.js: 工具函数
- src/api.js: API 调用
```

✅ **正确做法**：
跳过文件级别的总结，直接进入架构思想的分析。识别核心抽象，而不是列举文件。

### 2. API 功能罗列
❌ **错误示例**：
```markdown
## 功能特性
- 支持用户注册
- 支持数据查询
- 支持批量删除
```

✅ **正确做法**：
分析实现这些功能的核心数据结构和设计决策，而不是罗列功能。

### 3. "这个项目能干什么"的使用说明
❌ **错误示例**：
```markdown
## 使用方法
1. 安装依赖：npm install
2. 启动服务：npm start
3. 访问 http://localhost:3000
```

✅ **正确做法**：
聚焦于架构分析，而不是使用教程。如果必须提使用场景，仅用于说明架构决策的动机。

### 4. 没有代码支撑的空泛赞美
❌ **错误示例**：
```markdown
## 代码质量
这个项目的代码质量很高，架构设计很优雅。
```

✅ **正确做法**：
```markdown
## 设计美学
展示 10-30 行具体代码，然后从线性控制流、命名精确性、高内聚性等维度分析为什么美。
```

### 5. 过度使用类比的"文科生式"写作
❌ **错误示例**：
```markdown
这个架构就像炒菜一样，需要把各种食材按照正确的顺序放进去。
```

✅ **正确做法**：
使用精确的技术术语。如果必须使用类比，确保满足四个金线标准（见下文）。

## 禁止词汇（保留原克制原则）

以下词汇禁止使用：
- 本质
- 精妙
- 博弈
- 结晶
- 颠覆
- 革命性
- 前所未有
- 完美
- 极致
- 终极

**替代方案**：
- "本质" → "核心抽象"、"第一性原理"
- "精妙" → "优雅"、"精确"
- "博弈" → "权衡"、"取舍"
- "结晶" → "成果"、"产出"
- "颠覆" → "改进"、"优化"
- "革命性" → "突破性"、"创新性"
- "前所未有" → "独特"、"新颖"
- "完美" → "优秀"、"恰到好处"
- "极致" → "高度"、"深入"
- "终极" → "核心"、"关键"

## 类比使用四准则（金线标准）

**仅在满足以下全部条件时使用类比**：

### 1. 结构等价性
逻辑链路必须严丝合缝对应。

❌ **错误示例**：
"神经网络就像大脑一样工作"
- 问题：结构和逻辑不对等，这个类比不精确。

✅ **正确示例**：
"计算图的拓扑排序就像拆除建筑的逆序 - 你必须先拆除依赖别人的东西（ downstream 节点），才能拆除被别人依赖的东西（upstream 节点）。"
- 结构：计算图的依赖关系 ↔ 建筑的承重关系
- 逻辑：拓扑排序的逆序 ↔ 拆除的逆序

### 2. 认知杠杆率
用极低成本撬动极高深度。

❌ **错误示例**：
"这个数据结构就像一个抽屉，可以放东西"
- 问题：没有带来深度理解，只是表面的相似性。

✅ **正确示例**：
"哈希表的冲突解决就像停车场的车位分配 - 开放寻址法就像看到车位被占了就找下一个车位，而链地址法就像每个车位可以停多辆车（链表）。"
- 撬动的深度：立即理解两种冲突解决策略的本质差异

### 3. 生成预测力
读者能预判出项目的下一个特征。

❌ **错误示例**：
"这个架构很灵活"
- 问题：无法基于此做出任何预测。

✅ **正确示例**：
"micrograd 的标量优先设计意味着下一个优化方向很可能是向量化运算，而不是增加新的层类型。"
- 预测：可以预判作者的优化路径

### 4. 审美溢价
使用跨学科的高级概念。

❌ **错误示例**：
"这个就像搭积木"
- 问题：低级类比，没有审美价值。

✅ **正确示例**：
"这个依赖注入模式就像 category theory 中的 functor composition - 每个组件都是一个纯函数，依赖关系就是函数的组合。"
- 审美：引入数学概念，提升讨论的层次

## 最 Important 原则

### 如果逻辑已清晰，用术语替代类比

检查流程：
1. 先尝试用技术术语直接表达
2. 如果技术术语足够清晰，**删除类比**
3. 只有在技术术语无法表达或类比能显著提升理解时，才保留类比

### 自查清单

在使用类比前，问自己：
- [ ] 这个类比的逻辑链路是否严丝合缝？
- [ ] 这个类比是否用极低成本撬动了极高深度？
- [ ] 读者能基于这个类比预判出项目的下一个特征吗？
- [ ] 这个类比是否使用了跨学科的高级概念？
- [ ] 是否可以用技术术语替代这个类比？

如果第 5 项的答案是"能"，则删除类比。
